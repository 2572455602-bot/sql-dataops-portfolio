# 两分钟面试讲稿

## 开场

> 这是一个六表电商 SQL + DataOps 个人作品项目。先说明数据边界：当前外部样本没有可核验来源，结构和分布也疑似合成，所以系统把它标成 `PORTFOLIO / UNVERIFIED`；我不会说它来自淘宝、阿里或任何真实公司。

> 我把用户、商品、订单、行为和两张特征快照表，通过 SparkSQL 建成 RAW、ODS、DWD、DWS、ADS 五层数仓。订单和行为是两张独立事实表；特征快照只用于对账，Dashboard 的 KPI 都从清洗后的事实重新计算。

> 业务层输出完成订单样本金额、完成率、订单状态、每日趋势、小时行为、品类表现、五类客户分群，以及浏览到后续支付的用户级顺序漏斗。漏斗明确允许跨商品，所以我把它作为 SQL 时序能力证据，而不是营销归因。

> 工程上，每次运行记录六个输入文件和 SQL 的 SHA-256、run ID、行数、日期、Spark 版本、耗时与质量结果。20 项检查覆盖 Schema、主外键、金额、生命周期、层间行数和 ADS 指标对账；坏记录进 quarantine。只有 critical 全部通过才原子切换 Dashboard 的当前版本，失败批次不会覆盖上一版。

> 当前样本共 59,000 行，58,260 行有效，740 行因早于注册时间被隔离；质量结果是 16 pass、0 fail、4 warning。这个项目证明的是我的 SQL 建模、数据质量、可追溯发布、测试和 BI 交付能力，不是企业生产经验或真实经营成果。

## 常见追问

### 为什么 features 表不直接做 KPI？

> 因为它们是来源预计算结果，生成规则和时间点都不清楚。我把它们当 audit snapshot，用订单和行为事实独立聚合后对账；业务 KPI 仍由事实表重算，避免把来源派生错误直接传播到 Dashboard。

### 为什么有 warning 还允许发布？

> critical 表示会破坏结构、引用完整性、金额公式或业务对账的问题，必须阻断。warning 表示已知但可隔离或不参与指标的问题，例如注册前记录、`unit_price × quantity` 差异和 `999` 哨兵值。允许发布不等于隐藏风险，Dashboard 和 manifest 都会显示 warning。

### 完成订单样本金额是不是 GMV？

> 代码历史字段叫 `completed_gmv`，但对外我叫“完成订单样本金额”：已完成订单的 `actual_payment` 之和。源文件没有可验证的币种与会计语义，因此它不加货币符号，也不能代表真实公司的 GMV、收入或财务数字。

### 漏斗是否能证明商品转化？

> 不能。当前漏斗按同一用户的时间顺序连接浏览、点击、收藏/加购和后续支付订单，但不要求同一商品，也没有会话或归因窗口。因此它只能说明样本中的用户级顺序覆盖，不能证明因果或同商品转化。

### 质量通过能否证明数据真实？

> 不能。质量状态回答“数据是否满足项目契约并能安全发布”；来源状态回答“我们是否知道数据从哪里来”。两者是独立维度，所以这批数据即使 0 fail，仍保持 `UNVERIFIED`。

### 和企业级 DataOps 有什么差距？

> 我实现了契约、分层 SQL、幂等批次、质量门禁、manifest 和最后成功版本保护。企业还需要多源接入、云存储/湖仓、调度重试、Catalog、字段级血缘、权限、密钥、SLA、集中监控告警、多环境发布和成本治理。本项目证明我理解并能实现核心原则，但不冒充完整企业平台。

### 如果面试官要现场验证？

> 我会先展示极简首页和 Dashboard，再打开 manifest 与质量表，最后运行 `make demo` 或定向失败保护测试。评审者也可以从 ZIP 中运行 `make bootstrap`、`make demo`、`make test`、`make dashboard` 完整复现。
